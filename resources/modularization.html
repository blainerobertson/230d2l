Modularization

Introduction

Have you ever noticed that when you visit a website and move from page to page that there are certain areas on the page that don't change. To see what I mean let's visit subway.com (http://subway.com). Click around in the subway site. On each page scroll up and down and look for those parts of each page that don't really change. What did you find? If you said the top (header) and bottom (footer) you are very observant. These common parts are highlighted in figure 1:

Benefits

It is these common parts of a web page that can be reused. We call snippets of code that make up these reusable sections of the web page "modules" and the process of implementing them into the page "modularization". The benefits of modularizing our websites is to:

  • Save time - when we build modules of common code we only have to do it once. This means less time writing the same code again and again.
  • Easy maintenance - By having this common code stored off in a module, when it comes time to make a change to existing code, add new code or delete code, we only have to do it once for it to take effect throughout all the pages that use it.
  • Save Money - whether you're building a web site for yourself or someone else, the less time you have to spend building and maintaining it means less time and expense doing so. In business terms, it is being efficient.

Modules

You may be inclined to ask, "What exactly is a module?" The answer is, "A small amount of code, not stand-alone, that can be plugged-in to a larger collection of code in order to carry out or serve a particular task". Going back to Subway.com's header, you will notice that on the left is the logo, in the top right corner is a link to their international web site, to the left of the link are various tools, and at the bottom of the header is the main navigation bar. If you were to look at the source code for any of the pages you would find that the code used to build the header is about 100 lines long (alot of that is for the drop-down menus). But if you were to copy that code and paste it on a page of its own, it would look pretty strange. It is not meant to be used in that way, it is meant to be used at the top of each web page to add specific functionality to the page as a whole.

A simplified header example might lead us to create a code snippet that looks like this:

     <a href="/" title="Go to the subway.com home page">
          <img src="/images/subway-logo.jpeg" alt="subway.com logo">
     </a>
     <section id="tools"> ... code for various tools goes here ... </section>
     <nav id="page_nav">
          <ul>
               <li><a href="/">Home</a><li>
               <li><a href="/sandwiches">Sandwiches</a><li>
               <li><a href="/ingredients">Ingredients</a><li>
               <li><a href="/catering">Catering</a><li>
               <li><a href="/nutrition">Nutrition</a><li>
               <li><a href="/locations">Locations</a><li>
               <li><a href="/careers">Careers</a><li>
               <li><a href="/about">About Subway</a><li>
          </ul>
     </nav>

This code would be saved into a new text file, nothing else in the file except this code and given an appropriate name (e.g. header.php). The header.php file would be stored in a folder with other modules, so the storage folder could be named "modules" or something similar.

Implementing the Module

Having planned out and constructed the individual modules its now time to put them into effect. In our example we will be using a server-side scripting language called "PHP". Remember that the module is "plugged-in" to a larger collection of code. In our case that larger collection of code would be the <header> of a web page. To keep things simple only the header is shown, but it would be preceded by the opening code of the web page and followed by the content and other portions of the page. See the example code segment under The Result below.

An explanation of the code:

  • <?php ?> - this a PHP code block. By creating it and placing the PHP code within it, the server knows that the PHP must be executed before the web page can be returned to the browser.
  • include - this is a PHP function that follows the path to a module in order to draw it in and make it a part of the larger code collection at the point where the include function is written.
  • $_SERVER['DOCUMENT_ROOT'] - this is a PHP "Super Global" object and its method that instructs the server to provide a path from the root of the server's hard drive all the way to the root of the current website. Some web hosts do not require that you use this in order to make modules work, but it doesn't hurt to have it.
  • .(period) - a php concatenation symbol. In short, it takes the group of characters on the left (provided by $_SERVER['DOCUMENT_ROOT']) and joins them to the group of characters on the right ('/modules/header.php').
  • >'/modules/header.php' - this is the path from the root of the website to the module to be included. Please notice the single quote marks on either side. The single quotes mark this collection of characters as a "string". This is programming terminology for a collection of alpha-numeric characters.
  • When placed together, everything to the right of the word "include" is a path (an absolute path if you want to be technical) that tells the server exactly where to find the file that is to be included at this location.

Note! The web page where this PHP include is written, would itself have to have a .php extension. Say, for example, the web page were named "index", then it would have to be "index.php". It could not have any other extension if we want the include to work.

The Result

Since PHP is run on the server, here is how all of this code would work. Someone would see a link to index.php and click it. When the server got the request for index.php it would see the .php extension and ask the PHP engine to process it. The PHP engine would begin reading the code in the page looking for a PHP code block - <?php ... ?>. When it found one, it begins to execute the various commands inside of it. In our case, it would attempt to find the code in the module and include it in the <header>. Assuming that everything worked as it was supposed to, the header would look like this when the PHP engine was done:

<header id="page_header">
     <a href="/" title="Go to the subway.com home page">
          <img src="/images/subway-logo.jpeg" alt="subway.com logo">
     </a>
     <div id="tools"> ... code for various tools goes here ... </div>
     <nav id="page_nav">
          <ul>
               <li><a href="/">Home</a><li>
               <li><a href="/">Sandwiches</a><li>
               <li><a href="/">Ingredients</a><li>
               <li><a href="/">Catering</a><li>
               <li><a href="/">Nutrition</a><li>
               <li><a href="/">Locations</a><li>
               <li><a href="/">Careers</a><li>
               <li><a href="/">About Subway</a><li>
          </ul>
     </nav>
</header>

What is neat about using this technique, besides those mentioned earlier, is that when the server delivers index.php back to the requesting web browser is that there is no sign left of any of the PHP code block or any of the code inside of it. All that is left is the finished product. Think of it like ordering a sandwich at a restaurant. You order the sandwich and that is what is delivered to you. You don't see the preparation of it, the freezers where things are stored, the cutting board or utensils used to create it.

Make a Change

Reflect back on the list of benefits of modularization. Let's suppose that the folks at Subway decide to make soup a big part of their menu. Wow, we need to update that list of items in the navigation list. No problem. You open the header.php module and add the following code to the unordered list: <li><a href="/soup">Soups</a></li>.

Save the file and upload it to your website. Viola! You now have a soup navigation menu throughout your website - even if the site had 10,000 pages. Pretty Cool, huh?!

Summary

Modularization is a great technique for writing and storing small collections of code that represent reusable portions of a website into external files. Those files are then included using PHP or some other scripting language. The resulting file is then delivered to the requesting user-agent (usually a web browser) in its finished form, leaving no evidence of the scripting language. The key is to use modules with those portions of a webpage that stay the same from page to page to page. Using this technique saves time in both construction and maintenance and money.

Application

As you prepare to build web pages, it is helpful to build template(s). A template is simply a standard page (without content) that can be reused when a new content page is needed. The template(s) should include semantic structures with modularized common content as discussed in this activity. Use the steps outlined in this article to do so. Then, be sure to test it on the actual web server to insure that things are working as they should.
